## 1. 目标与优先级

构建一个**以 A 股为重点**的行情看板 + 多智能体交易决策系统,前后端分离。

**优先级(用户明确)**:
1. **模拟交易**(最高)— 一个跟随真实行情的模拟账户
2. **机会发现**— 行业热点、龙头公司、潜在爆发候选股
3. 研报/财报的 AI 分析作为决策支撑

**定位**: B 类「量化策略开发平台」为主、A 类「个人研究工具」为辅。个人使用,不接实盘。

### 诚实约束(写进设计的前提)
- 「预测未来爆发股」不可靠。系统只做**结构化线索发现**(资金流、研报评级变化、财报超预期、量价异动、行业景气),判断权始终留给用户。不做「看起来很准」的伪精确产品。
- 模拟交易**净值每日盯市结算**(用户选择 B):每日收盘后用 tushare 当日数据更新持仓市值与净值曲线。用户有 tushare 高积分,数据层无忧。
- **决策/调仓为周频**(见 §2),与每日盯市分离:周频出建议、每日算净值,两者不冲突。

### 量化数据正确性(用户评审补充,硬性要求)
- **历史数据构建是前置硬任务**(切片1):首次需将 N 年历史日线灌入 qlib 建库,之后增量更新。无历史数据则回测/因子/量价异动信号无法成立。
- **复权处理**:tushare `pro_bar` 支持 `adj=None/qfq/hfq`。**存储口径 = 不复权原始 OHLCV + `adj_factor`(复权因子)**,前复权(qfq)/后复权(hfq)按需推导。绝不只存单一复权价,否则因子与回测会系统性出错。这是 qlib 的标准数据模型。
- **API 用量控制**:发现引擎打分排序后**硬性截断 Top-N**(默认 8 只)才进决策层,绝不把整个候选池喂给 `propagate()`;DeepSeek 调用做节流/批控。

## 2. 关键技术决策(已确认)

| 决策点 | 选择 | 理由 |
|---|---|---|
| 净值结算粒度 | 每日盯市(日线收盘) | 净值曲线平滑,不因周频调仓断裂 |
| 决策/调仓频率 | **周频**(默认周五收盘跑→下周一建议),可配置缩短 | 用户要求;省 API;符合中线持仓 |
| 候选→决策截断 | **Top-N=8**(可配)进决策层 | 控制 DeepSeek/Agent API 用量 |
| 复权口径 | 存不复权 OHLCV + `adj_factor`,qfq/hfq 按需推导 | 量化正确性;qlib 标准 |
| 历史数据 | 切片1 前置:N 年历史灌 qlib 建库 + 增量 | 回测/因子/异动信号的基础 |
| 下单方式 | 人机协同 | Agent 出建议清单,用户拍板,采纳才进模拟账户 |
| 看板形态 | 前后端分离 Web 应用 | 用户明确要 React + FastAPI |
| 后端 | Python + FastAPI | tushare/qlib/scrapling/tradingagents 全是 Python 生态 |
| 前端 | React + Vite + ECharts | 用户指定 React;ECharts 适合行情图 |
| 决策引擎 | 复用 TradingAgents-CN 的 agent 内核作依赖 | 已适配 A 股 + tushare + DeepSeek,避免重写最难部分 |
| LLM | DeepSeek `deepseek-v4-pro`(1M 上下文,OpenAI 接口兼容) | 用户指定;1M 上下文适合整篇研报财报 |
| 量化数据仓库 | qlib(量价/因子) + PostgreSQL(应用数据) | qlib 当「量化数据集」用,不当通用数据库 |
| 爬虫 | scrapling | 只抓 tushare 给不了的:研报全文、财报原文、新闻舆情 |

### TradingAgents 集成方式(核心)
GitHub 上有两个相关项目:
- **`TauricResearch/TradingAgents`**(原版):LangGraph 多智能体,`propagate(ticker, date)` → 五档评级 + 推理。数据走 yfinance,A 股不可用。
- **`hsliuping/TradingAgents-CN`**(中文增强版):原生 A 股(tushare/akshare/baostock)、支持 DeepSeek、已有 FastAPI + Vue3 + 模拟交易 + 选股筛选。

**决策(用户选 A)**: 仅复用 CN 版的 `tradingagents` agent 内核作为**决策库依赖**,外层(React 应用、发现引擎、qlib、模拟交易)全自建。既尊重 React + FastAPI 的要求,又不重写最难的「A 股多智能体数据适配」。**不**整体吃 CN 版的 Vue/MongoDB/Redis 全家桶。

## 3. 系统架构

前后端分离,后端六层,各层单一职责、接口隔离。

```
┌─────────────────────────── 前端 (React + Vite + ECharts) ───────────────────────────┐
│  行情看板 · 持仓/净值 · 候选池 · 每日建议清单(采纳/否决) · 个股详情(研报财报摘要)   │
└──────────────────────────────────────┬───────────────────────────────────────────────┘
                                        │ REST / SSE
┌──────────────────────────────────────▼─────────────── 后端 (FastAPI) ─────────────────┐
│                                                                                        │
│  ① 数据接入层    tushare(行情/财务/资金流) + scrapling(研报/财报/新闻)  ←可替换接口   │
│  ② 数据仓库层    qlib(量价/因子)  +  PostgreSQL(持仓/成交/建议/分析结论/文本)         │
│  ③ 分析层        DeepSeek V4-Pro 读研报财报 → 结构化结论                                │
│  ④ 发现引擎      多信号打分器(可插拔信号源)→ 候选池                                    │
│  ⑤ 决策层        TradingAgents-CN 内核:候选股 → propagate() → 买卖建议 + 风控          │
│  ⑥ 模拟交易引擎  日线结算、持仓、成交、净值曲线                                          │
│                                                                                        │
│  ⑦ 调度器        APScheduler:收盘后批处理流水线                                        │
└────────────────────────────────────────────────────────────────────────────────────┘
```

### 各层职责与接口

- **① 数据接入层** — `MarketDataSource` 接口(可替换实现:tushare / 未来 L2)。爬虫 `ReportCrawler`(scrapling)抓研报/财报/新闻文本。
- **② 数据仓库层** — qlib 存量价与因子(给回测/因子);PostgreSQL 存应用态数据(账户、持仓、成交、建议、分析结论、爬取文本、候选池快照)。
- **③ 分析层** — `ResearchAnalyzer`:输入研报/财报原文 → DeepSeek → 输出结构化 JSON(评级与变化、是否超预期、关键数字、风险点、摘要),入库。
- **④ 发现引擎** — `DiscoveryEngine`:**多信号打分器**,信号源实现统一 `SignalProvider` 接口。输出**候选池**(含分项得分与理由)。
- **⑤ 决策层** — `DecisionService`:封装 `TradingAgentsGraph.propagate(code, date)`,注入 DeepSeek 配置与(可选)分析层结论作为额外上下文;输出标准化决策对象(方向/评级/理由/风控)。
- **⑥ 模拟交易引擎** — `PaperBroker`:管理模拟账户。采纳的建议 → 次日开盘价成交 → 每日收盘结算盈亏、更新持仓、记成交、算净值曲线。
- **⑦ 调度器** — 收盘后按序触发流水线(见 §4)。

## 4. 数据流(每日盯市 + 周频决策)

**A. 每日(每个交易日收盘后)— 数据维护 + 盯市**
```
  1. tushare 拉当日行情/财务/资金流  → 增量灌 qlib + PostgreSQL(不复权 OHLCV + adj_factor)
  2. 重算持仓市值 → 更新净值曲线(每日盯市,⑥)
```

**B. 每周(默认周五收盘后)— 发现 + 决策**
```
  3. scrapling 抓本周研报/财报/新闻  → DeepSeek 分析(③) → 结构化结论入库
  4. 发现引擎(④)汇总所有信号源打分 → 候选池
  5. ★ 截断 Top-N(默认 8)        → 仅这 N 只进决策层(控 API 用量)
  6. 决策层(⑤)逐只 propagate     → 「下周建议清单」(买/卖/调仓 + 理由 + 风控)
  7. 推送摘要提醒用户(微信 / 前端通知)
```

**C. 下一交易日开盘 — 执行**
```
  8. 用户已采纳的建议 → 模拟账户按开盘价成交(⑥),之后回到每日盯市
```

> 频率与 Top-N 均可配置;「实在有必要可缩短决策频率」通过配置项实现,默认周频。
> 历史数据**首次全量灌库**是一次性前置任务(切片1),不在此循环内。

### 发现引擎与分析层的耦合(关键澄清)
发现引擎是「多信号打分器」,分析层(scrapling+DeepSeek)是它的**信号源之一**,而非独立模块:

- **量化信号(纯 tushare)**:资金流入、行业景气(热点);市值+基本面(龙头);量价异动;**财报超预期(数字层面:营收/利润同比、是否超一致预期)**。
- **质化信号(scrapling + DeepSeek)**:**研报评级上调/下调**、研报/财报文字洞察(风险点、超预期原因、管理层口径)。

分析层的结构化结论有三个下游:
1. 作为新 `SignalProvider` **注册进发现引擎** → 候选池打分更准;
2. 作为额外上下文**喂给决策层的 TradingAgent**;
3. 展示在**看板个股详情页**。

因为发现引擎从一开始就按「信号可插拔」设计,质化信号是后插的一根信号线,**不需返工**。

## 5. 数据模型(PostgreSQL,初版要点)

- `accounts` — 模拟账户(初始资金、现金、总资产)
- `positions` — 持仓(代码、数量、成本价、市值)
- `trades` — 成交流水(代码、方向、价格、数量、时间、来源建议id)
- `equity_curve` — 每日净值快照(每日盯市写入)
- `suggestions` — 周频建议(代码、方向、理由、风控、Agent原始输出、状态:待定/采纳/否决、决策周)
- `candidates` — 候选池快照(代码、综合分、分项信号分、是否入选 Top-N、日期)
- `research_insights` — 分析层结构化结论(代码、来源文档、评级与变化、超预期标记、关键数字、风险点、摘要、原文链接)
- `raw_documents` — 爬取的研报/财报/新闻原文与元数据

量价数据存 qlib 自有格式(**不复权 OHLCV + adj_factor**),不入 PG;qfq/hfq 推导后供回测/展示。复权因子也可在 PG 留一份便于查询。

## 6. 构建顺序(垂直切片)

原则:不六层齐上;先让能端到端跑通的最小骨架活起来。每片独立可交付,各自 spec → plan → build。

- **切片 0 — 项目骨架**
  Monorepo(`backend/` FastAPI + `frontend/` React+Vite)、安装 **Python 运行环境**、引入 `tradingagents` 依赖、`.env`(tushare token、DeepSeek key)、PostgreSQL、GitHub 仓库同步。
- **切片 1 — MVP 端到端(最关键)**
  - **1a 历史数据建库(前置)**:tushare 拉 N 年历史日线(不复权 OHLCV + adj_factor)→ 灌 qlib 建库;校验复权推导正确(qfq/hfq 拼接无跳变);建立每日增量更新任务。
  - **1b 模拟交易引擎**:每日盯市结算、持仓、成交、净值曲线 + 最简 React 看板(手动下单、看持仓盈亏)。
  - 先让历史数据就位、模拟账户真正跑起来。
- **切片 2 — 发现引擎**
  按可插拔信号接口搭好,先接量化信号(资金流/景气/市值基本面/量价异动/财报超预期)→ 候选池 + **Top-N 截断** + 看板展示。
- **切片 3 — 接入 TradingAgent 决策**
  Top-N 候选 → `propagate(code, date)`(DeepSeek,周频)→ 建议清单 + 人机协同采纳 → 进模拟账户。API 用量节流。
- **切片 4 — 研报/财报分析**
  scrapling 爬取 → DeepSeek V4 分析 → 结构化结论入库;**注册为发现引擎新信号源** + 喂决策层 + 个股详情页展示。
- **切片 5 — 增强**
  qlib 因子/回测、微信收盘摘要推送、看板美化。

**首批实现:切片 0 + 1。**

## 8. 风险与开放问题

- **qlib 安装与数据维护成本**:qlib 对 Python/依赖版本敏感,首次落地需 spike 验证。
- **scrapling 稳定性与合规**:爬研报/财报源站有反爬与合规风险;做成可降级(失败不阻塞主流水线)。
- **DeepSeek 成本**:整篇研报财报进 1M 上下文,需监控 token 用量;质化信号按需触发,非全量。
- **TradingAgents-CN 内核耦合**:作为依赖引入时需确认其包结构能被单独 import(不强依赖其 Vue/Mongo 部分);切片 3 前需 spike 验证。
- **数据库选型**:PG vs CN 版用的 Mongo——本设计用 PG(关系型更适合账户/成交)。✅ 用户评审无异议,已定。
- **复权与历史数据**:✅ 用户评审补充已纳入(§1/§2/§6 切片1a)。存不复权+adj_factor;历史建库为切片1 前置;决策周频、净值每日盯市;Top-N=8 截断控 API。
